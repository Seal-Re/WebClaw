## 一、设计目标与原则

### 1.1 业务目标

- 在既定授权范围内实现**决策—执行—验证—留痕**的自动化安全测试闭环。
- 将分散的工具能力、测试策略与验证流程编排成**可持续、可复用、可度量**的工程化能力，支撑整改闭环与审计要求。

### 1.2 强制执行原则（最高优先级）

| 原则 | 说明 |
|------|------|
| **分块设计** | 模块划分清晰、职责单一，支持跨模块协作开发。 |
| **解耦部署与测试** | 各模块通过标准接口或 Mock 数据可独立进行单元测试与功能验证，不依赖其他模块实现。 |
| **奥卡姆剃刀** | 剔除非必要冗余组件与流转环节；单一事实来源，其余为派生或视图。 |
| **完备性自检** | 设计需支撑多阶段攻击、过程留痕、双平台、全流程覆盖及企业级报告等核心业务闭环。 |

### 1.3 技术栈与差异化

- **大模型**：支持 GPT、Claude、DeepSeek 等主流接口，用于理解、推理与决策。
- **知识体系**：以 **OpenClaw** 替代传统爬虫，实现全天联网搜索→筛选→写入知识库，并与在线 RAG 分离部署、统一出口。
- **编排**：状态机 + 阶段内任务表为唯一事实来源；攻击图为静态规范，重复防控为输入增强，不引入第二套状态。

## 二、赛题与需求对照

### 2.1 问题期望覆盖

| 问题期望 | 设计对应 |
|----------|----------|
| ① 基于大模型的自动化安全测试系统，智能化与自动化 | Orchestrator + LLM Decision Engine 驱动全流程；决策—执行—验证—留痕闭环。 |
| ② 大模型理解与推理，提高准确性与一致性 | LLM 规划与决策；RAG 提供知识增强；目标识别与策略选择显式化。 |
| ③ 降低对人工经验的依赖，提升效率与覆盖 | 知识模块 [(OpenClaw),(RAG)] + 多 Skill 覆盖全流程。 |
| ④ 支持 Windows + Linux 双平台，自动识别目标并选择策略 | Target Context 含目标类型/OS；LLM 据此选策略与 Skill；灰盒上下文支持。 |
| ⑤ 集成多款安全工具，覆盖网络扫描、漏洞扫描、Web、暴力破解、后渗透 | Skill Executor 按类别注册；见 [§7](#七skill-分类与工具覆盖)。 |
| ⑥ 可演示、可扩展的系统原型 | 模块化 + 标准接口；新工具=新 Skill，新漏洞=知识库扩展。 |

### 2.2 任务要求与技术要求

- **清晰架构、模块化/智能体协作**：五类模块 + Knowledge 内双服务 [(OpenClaw),(RAG)]，职责清晰。
- **大模型分析与决策**：目标分析→漏洞识别→利用→结果验证，由状态机 + LLM + Skill 实现；显式包含「目标识别与策略选择」环节。
- **完整流程 + 结构化报告**：Trace & Report 负责事件流与报告；报告含发现、复现步骤、证据、修复建议、合规映射。
- **可扩展性**：新工具通过 Skill 注册；新漏洞/CVE 通过 OpenClaw 采集与 RAG 检索接入。
- **流程控制与可视化支撑**：事件流与报告数据结构化，可支撑后续前端可视化（本设计不包含前端）。

### 2.3 量化指标支撑

| 指标 | 架构支撑 |
|------|----------|
| 漏洞检测率/误报率/CVE 覆盖 | Knowledge(RAG) + LLM 决策 + Skill 执行 + 结果校验。 |
| 双平台 | Target Context 存 OS/类型；LLM 选策略；Skill 覆盖 Win/Linux 工具。 |
| 工具数 ≥30/≥50 | Skill Executor 注册与分类；见 §7。 |
| 单目标时间/并发 | 同步/异步划分 + 超时；Orchestrator 任务调度与多目标。 |
| 多阶段链式 | 状态机 + 阶段内任务表 + 攻击图规范。 |
| 详细报告+修复建议 | Trace & Report 结构 + RAG 引用；见 §8。 |

### 2.4 与 PTES 七阶段映射

| PTES 阶段 | 本平台阶段/说明 |
|-----------|-----------------|
| Pre-engagement | 任务创建、范围与授权配置（Target Context / 任务参数）。 |
| Intelligence Gathering | 侦察：端口/服务/资产发现。 |
| Threat Modeling | 目标识别与策略选择：推断 OS/类型，选择测试策略。 |
| Vulnerability Analysis | 漏洞识别：扫描与漏洞分析。 |
| Exploitation | 利用：PoC/利用尝试。 |
| Post-Exploitation | 后渗透：权限维持、横向移动等（可选）。 |
| Reporting | 报告生成：执行摘要、发现、复现、修复建议、合规映射。 |

## 三、最终模块拓扑与职责

### 3.1 模块拓扑图

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           Orchestrator（编排器）                              │
│  状态机 + 阶段内任务表（唯一事实来源）· 同步/异步划分 · 超时· 多目标调度       │
└───────┬─────────────────┬─────────────────┬─────────────────┬─────────────────┘
        │                 │                 │                 │
        ▼                 ▼                 ▼                 ▼
┌───────────────┐ ┌───────────────┐ ┌───────────────┐ ┌───────────────┐
│ Knowledge     │ │ LLM Decision  │ │ Skill         │ │ Trace &       │
│ Module        │ │ Engine        │ │ Executor      │ │ Report        │
│ [(OpenClaw),  │ │ 规划/决策/解析 │ │ 原子化执行    │ │ Service       │
│  (RAG)]       │ │ 目标类型→策略  │ │ + 沙箱(可选)  │ │ 留痕+报告生成  │
└───────────────┘ └───────────────┘ └───────────────┘ └───────────────┘
        │                 │                 │                 │
        └─────────────────┴────────┬────────┴─────────────────┘
                                    │
                     ┌──────────────┴──────────────┐
                     │  Target Context Store       │
                     │  目标/资产/OS类型/任务表   │
                     └────────────────────────────┘
```

### 3.2 各模块精准职责

| 模块 | 职责 | 不负责 |
|------|------|--------|
| **Orchestrator** | 维护状态机与阶段内任务表（唯一事实来源）；按攻击图规则校验迁移与依赖；同步/异步调用 LLM、RAG、Skill、Trace；多阶段与多目标调度；超时与重试；驱动「目标识别与策略选择」。 | 不做具体漏洞判断、不执行工具、不生成报告正文。 |
| **Knowledge Module** | **(OpenClaw)** 替代爬虫：全天联网搜索→筛选→写入知识库；**(RAG)** 为运行中任务提供检索与上下文增强；共享知识库存储。 | 不决策、不执行；与单次任务解耦的写库由 (OpenClaw) 独立调度。 |
| **LLM Decision Engine** | 接收阶段、目标上下文（含 OS/灰盒）、知识片段、历史、重复警告；输出下一步动作（Skill 调用、参数、下一阶段或结束）；解析 tool call。 | 不直接调工具、不持久化轨迹、不维护状态。 |
| **Skill Executor** | 注册并执行原子化 Skill；可选沙箱（接口与实现解耦）；统一入参/出参；覆盖网络扫描、漏洞扫描、Web、暴力破解、后渗透等类别。 | 不做阶段规划、不查知识库、不生成报告。 |
| **Trace & Report Service** | 按任务收集事件流；生成企业级报告：执行摘要、发现、复现步骤、证据、风险等级、修复建议（可引用 RAG）、合规映射。 | 不参与决策、不执行工具。 |
| **Target Context Store** | 存储目标信息、资产、OS/系统类型、灰盒字段、阶段输出、漏洞摘要、任务表缓存等，供编排与 LLM 使用。 | 仅读写上下文，无业务逻辑。 |

### 3.3 攻击图与重复防控的定位（奥卡姆收敛）

- **攻击图**：作为**静态规范**定义阶段、迁移条件、阶段内子任务依赖；运行时**不**维护图状态，Orchestrator 按图规则校验「当前迁移是否合法」「阶段内依赖是否满足」。
- **重复防控**：**无独立状态**；轻量逻辑（可在 Orchestrator 内或独立服务）消费 Trace 近期事件，计算行动指纹/相似度，将「重复警告」注入 LLM 输入；迁移与调用仍完全由 Orchestrator 根据 LLM 输出执行。

## 四、知识模块：[(OpenClaw), (RAG)]

### 4.1 定位说明

- **一个模块、两个服务**，共享同一知识库；(OpenClaw) 与 (RAG) 职责分离、接口独立，可单独测试与扩展。
- **离线知识库**含义：用 OpenClaw **替代原有爬虫**，全天联网搜索与检索相关知识，筛选后加入知识库；与「断网」无关，与「写库流程相对独立于单次任务」有关。

### 4.2 结构示意

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     Knowledge Module（知识模块）                           │
│   ┌─────────────────────────────┐    ┌─────────────────────────────┐   │
│   │  (OpenClaw)                  │    │  (RAG)                      │   │
│   │  知识获取 · 替代爬虫          │───▶│  在线检索 · 为 Agent 提供   │   │
│   │  全天联网搜索→筛选→写库       │    │  增强片段 · 多模态/溯源     │   │
│   └─────────────────────────────┘    └─────────────────────────────┘   │
│              │                                      ▲                    │
│              ▼                                      │ 检索请求            │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │                    共享知识库存储                                  │   │
│   └─────────────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────────────┘
```

### 4.3 (OpenClaw) 知识获取服务 — 增强

- **多源采集**：除通用搜索外，定向采集 CVE/PoC、HackTricks、Metasploit 模块说明、OWASP 条目等；筛选后写入统一知识库；支持按类型/标签（CVE、Web、后渗透等）打标。
- **结构化与可执行片段**：对部分条目做轻量结构化（CVE ID、CVSS、影响产品、复现步骤、修复建议）；可支持「战术片段」（如先 nmap 再 msf 某模块）以固定 schema 存入，供战术链与报告引用。
- **与 OpenClaw 生态对齐**：可封装 OpenClaw 的 kb-search / knowledge-vault 等能力，用于检索增强的采集策略（如按主题检索后再写入自有知识库），形成技术卖点。

### 4.4 (RAG) 在线检索服务 — 增强

- **多模态检索**：支持按类型/标签/CVE ID 过滤（如阶段为「漏洞利用」时优先拉取 PoC 与利用步骤）。
- **引用与溯源**：返回 chunk 带 `source`、`type`、`cve_id`（若有）；报告生成可将修复建议/复现步骤**引用到具体知识库条目**。
- **灰盒增强**：接收目标类型、已知服务/框架、已有资产等灰盒信息，作为检索条件或重排依据，提升策略与工具选择的针对性。

### 4.5 对外接口

- 其他模块**仅依赖 (RAG) 的检索接口**；(OpenClaw) 不对外提供业务 API，仅更新知识库。
- 检索请求：`task_id, phase, query_text, target_context_snapshot, grey_box_hints` → 响应：`chunks[]`（含 source, type, cve_id 等）。

## 五、编排层：唯一事实来源与同步/异步

### 5.1 唯一事实来源

- **Orchestrator** 维护：
  - **状态机**：阶段集合 + 允许的迁移规则（与 PTES 可映射）。
  - **阶段内任务表**：当前阶段下已提交的同步/异步任务及其状态（pending / running / done / failed / timeout）。
- 攻击图为静态规范；重复防控仅产生「重复警告」输入，不维护独立状态。

### 5.2 同步/异步边界（回应红队与效率指标）

| 类型 | 适用场景 | 行为 | 超时 |
|------|----------|------|------|
| **同步** | 单次、短时 Skill 调用（如单目标 Nmap、单 URL SQLMap） | Orchestrator 阻塞等待结果后继续 | 例如 2 分钟，超时标记失败，结果仍返回 |
| **异步** | 长时或批量任务（全段扫描、批量 PoC） | 提交到任务队列；Orchestrator 记录「本阶段已提交任务」，依「任务完成/超时」事件更新任务表 | 可配置 |

- **阶段完成语义**：仅当当前阶段所有任务到达终态（done / failed / timeout）后，才允许 LLM 驱动迁移到下一阶段；保证攻击链先后依赖（如先扫到 80 再测 Web）不被破坏。

### 5.3 流程概览

1. **初始化**：创建任务，写入 Target Context（初始目标、范围等）。
2. **侦察**：Orchestrator 调用 LLM + RAG，驱动侦察类 Skill；结果写入 Target Context。
3. **目标识别与策略选择**：根据 Target Context（含侦察结果、灰盒）请求 LLM 推断 OS/类型并选择策略；结果写入上下文。
4. **漏洞识别**：RAG + LLM 决策调用漏洞扫描/检测类 Skill；结果写入并进入利用阶段。
5. **利用与验证**：LLM 决策调用利用/验证类 Skill；Trace 记录每次调用与结果。
6. **报告**：Orchestrator 触发 Trace & Report 生成报告（含修复建议、合规映射）；修复建议可引用 RAG 来源。

## 六、模块间标准通信接口

### 6.1 设计原则

- 所有跨模块交互通过**契约（接口/事件/消息）**完成，无直接依赖实现；任一模块可被 Mock 或替换。
- 所有跨模块 API 具备 **OpenAPI 3.0 或 JSON Schema**；Mock 数据与 CI 契约测试必须通过 schema 校验；契约由**接口提供方**维护并版本化。

### 6.2 核心契约摘要

| 调用方 → 被调方 | 请求要点 | 响应要点 |
|-----------------|----------|----------|
| Orchestrator → (RAG) | task_id, phase, query_text, target_context_snapshot, grey_box_hints | chunks[]（content, source, type, cve_id, score） |
| Orchestrator → LLM Decision Engine | task_id, phase, current_state, task_table_summary, target_context, knowledge_chunks, history_summary, repetition_warning | action_type, skill_id, params, reasoning；可选 recommended_phase |
| Orchestrator → Skill Executor | task_id, skill_id, params, timeout_ms, sandbox_level | success, stdout, stderr, artifacts, exit_code, duration_ms |
| 各模块 → Trace | 事件流 | 无响应（Trace 仅消费） |
| Report 生成请求 → Trace & Report | task_id, template_id, options | report_id, status, download_url_or_content |

### 6.3 事件模型（Trace）

- 事件格式：`task_id, timestamp, event_type, phase, payload`。
- event_type 示例：phase_start, phase_end, skill_invoked, skill_result, task_submitted, task_completed, task_timeout, llm_decision, error。
- 异步任务生命周期（提交→运行中→完成/超时）均有明确事件类型与 payload schema，便于基于事件序列的集成测试与复现。

### 6.4 依赖方向

- Orchestrator 依赖各模块的「接口」；Knowledge、LLM、Skill、Trace 之间无直接依赖，仅通过 Orchestrator 串联。

## 七、Skill 分类与工具覆盖

### 7.1 类别与赛题对应

| 类别 | 说明 | 示例工具/能力 |
|------|------|----------------|
| 网络扫描 | 主机/端口/服务发现 | Nmap、Masscan 等 |
| 漏洞扫描 | 漏洞与 CVE 检测 | 各类漏洞扫描器、CVE 检测脚本 |
| Web 应用测试 | Web 漏洞与注入 | SQLMap、目录/敏感文件探测、XSS/CSRF 等 |
| 暴力破解 | 弱口令、爆破 | 针对服务/Web 的爆破工具 |
| 后渗透 | 权限维持、横向移动 | 与靶机环境匹配的后渗透模块 |

### 7.2 扩展与数量

- 工具总数满足基础 ≥30、进阶 ≥50；新工具通过注册新 Skill、暴露统一入参/出参接入。
- Skill Executor 对外仅暴露「执行请求/结果」接口；**沙箱为执行器的一种实现**；单测使用 Stub 实现；真实沙箱隔离性在具备隔离环境的流水线中验证。

## 八、企业级报告与合规

### 8.1 报告结构

- **执行摘要**：范围、结论、主要风险、关键发现条数。
- **方法论**：映射到 PTES 七阶段及本平台阶段名。
- **发现列表**：每项含标题、描述、复现步骤、影响、风险等级（高/中/低或 CVSS）、修复建议、**引用来源**（RAG 的 source/type/cve_id）。
- **附录**：工具列表、证据索引（Trace artifact_id）、可选截图/输出摘要。

### 8.2 风险与合规

- **风险定级**：来自 LLM 或规则；若识别到 CVE，可从知识库或规则表取 CVSS。
- **合规映射**：为发现打标签，如 OWASP Top 10、NIST 控制、等保 2.0/3.0 相关条款；数据可从 RAG chunk 元数据或规则表获取。
- **修复建议**：优先从 RAG 返回的 chunk 生成，并保留「引用知识库条目」，满足可追溯与审计。

## 十一、技术评审结论与妥协要点

### 11.1 红队关切与妥协

- **问题**：沙箱下工具执行效率与阻塞/异步影响攻击链连贯性与单目标时间。
- **结论**：显式区分同步（短时+超时）与异步（长时/批量）；阶段完成语义为「本阶段所有任务终态后才迁移」；攻击链依赖由状态机 + 任务表 + 攻击图规则保证。

### 11.2 架构师关切与妥协

- **问题**：攻击图 + 状态机 + 重复检测多源状态，违反奥卡姆剃刀并可能状态爆炸。
- **结论**：唯一事实来源为 Orchestrator（状态机 + 阶段内任务表）；攻击图为静态规范；重复防控为输入增强、不持状态。

### 11.3 QA 关切与妥协

- **问题**：契约无人维护则 Mock 与实现易脱节；沙箱在 CI 无法真实验证。
- **结论**：所有跨模块 API 具备书面 schema；Mock 与 CI 契约测试通过 schema 校验；Skill Executor 与沙箱实现解耦，单测用 Stub，沙箱隔离性在合适环境中单独验证并文档化。

### 11.4 最终架构原则小结

- **单一事实来源**：编排状态仅存在于 Orchestrator（状态机 + 任务表）。
- **同步/异步显式划分**：短时同步+超时，长时异步+事件驱动；阶段完成语义清晰。
- **可测性**：契约 + 分层测试（Stub 单测、真实沙箱在合适环境）；异步通过事件契约可复现。

## 十二、核心通信契约与数据结构 (MVP阶段定死)

### 12.1 设计原则与约束

- **接口定死，实现延后**：跨模块交互严格遵循本节定义的 RESTful API 与 JSON Schema；底层框架与数据库可后选，内部逻辑可先用 Mock 实现。
- **极简主义**：不设计过度复杂的鉴权或非核心字段，仅保留支撑「决策—执行—留痕」主链路的最少必要字段。
- **开发顺序**：优先实现 **12.2～12.4** 三条核心接口，达成最小可跑通（决策→执行→留痕）；**12.5、12.6** 为附加接口，在完整 MVP 阶段再接上知识检索与报告生成。

---

### 12.2 编排器请求决策 (Orchestrator → LLM Decision Engine)

**业务场景**：编排器将当前目标状态、所处阶段、可用工具列表发送给 LLM 引擎，请求推断下一步动作（执行某 Skill、进入下一阶段或结束任务）。

| 项目 | 说明 |
|------|------|
| **Endpoint** | `POST /v1/decide` |
| **HTTP 方法** | POST |

**请求体 (Request Body)**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| task_id | string | 是 | 任务唯一标识 |
| current_phase | string | 是 | 当前阶段，如 RECON, VULN_SCAN, EXPLOIT, REPORT |
| target_context | object | 是 | 当前目标上下文（已发现端口、服务、资产等） |
| history_summary | string | 是 | 近期步骤摘要，供 LLM 避免重复 |
| available_skill_ids | string[] | 否 | 当前可用的 skill_id 列表，缺省表示不限制 |

**请求示例**

```json
{
  "task_id": "task-2026-001",
  "current_phase": "RECON",
  "target_context": {
    "target": "192.168.1.100",
    "open_ports": [22, 80, 443],
    "services": ["ssh", "http", "https"],
    "os_hint": "Linux"
  },
  "history_summary": "已完成端口扫描；发现 80 开放，未进行 Web 扫描。",
  "available_skill_ids": ["nmap", "nikto", "sqlmap"]
}
```

**响应体 (Response Body)**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| action_type | string | 是 | EXECUTE_SKILL \| NEXT_PHASE \| FINISH |
| skill_id | string | 条件 | action_type 为 EXECUTE_SKILL 时必填 |
| params | object | 条件 | 执行 Skill 时的参数，与 skill_id 配套 |
| next_phase | string | 条件 | action_type 为 NEXT_PHASE 时必填 |
| reasoning | string | 否 | 简短决策理由，便于留痕与调试 |

**响应示例**

```json
{
  "action_type": "EXECUTE_SKILL",
  "skill_id": "nikto",
  "params": {
    "target": "http://192.168.1.100",
    "port": 80
  },
  "reasoning": "80 端口开放 HTTP，建议进行 Web 漏洞扫描。"
}
```

### 12.3 编排器下发执行指令 (Orchestrator → Skill Executor)

**业务场景**：编排器解析 LLM 决策后，向工具执行器（沙箱）下发具体的扫描或漏洞利用指令（如 Nmap、SQLMap）。

| 项目 | 说明 |
|------|------|
| **Endpoint** | `POST /v1/execute` |
| **HTTP 方法** | POST |

**请求体 (Request Body)**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| task_id | string | 是 | 任务唯一标识 |
| skill_id | string | 是 | 工具标识，如 nmap, sqlmap |
| target | string | 是 | 目标地址（IP 或 URL） |
| params | object | 否 | 该 Skill 的扩展参数，键值由 skill 定义 |

**请求示例**

```json
{
  "task_id": "task-2026-001",
  "skill_id": "nmap",

```

**响应体 (Response Body)**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|

| raw_stderr | string | 否 | 原始终端标准错误 |

**响应示例**

```json
{

  "parsed_artifacts": {
    "open_ports": [22, 80, 443],

      { "port": 22, "name": "ssh" },
      { "port": 80, "name": "http" },
      { "port": 443, "name": "https" }

  },
  "raw_stdout": "Nmap scan report for 192.168.1.100\n...",

}
```

## 十二、性能与准确性优化策略

### 12.1 整体目标

- 在不改变「单一事实来源（Orchestrator + Target Context）」和模块职责边界的前提下，通过**编排策略与工具使用策略优化**，提升：
  - 单目标测试效率（时间、资源占用）；
  - 关键漏洞发现的**准确率**（减少漏报与误报）；
  - 系统在高负载、多目标场景下的可控性与可调优性。

### 12.2 阶段内并行与任务粒度

- **阶段内尽量并行**：在同一阶段（如 RECON / VULN_SCAN）中，对**彼此无前后依赖**的 Skill 采用异步任务提交，由 Orchestrator 通过「阶段内任务表」统一追踪：
  - 示例：在侦察阶段，可并行执行端口扫描（Nmap）、子域名枚举（Amass/Subfinder）、目录探测（Dirsearch）等。
  - Orchestrator 仅在 **所有子任务进入终态（done/failed/timeout）后** 才允许迁移到下一阶段。
- **合理控制任务粒度**：
  - 对长时/大范围扫描拆分为多个子任务（如按网段、端口段拆分），便于并行与重试。
  - 对非常轻量的操作可合并为单任务，避免过细粒度带来的调度开销。

### 12.3 规则与 LLM 的分工

- **规则兜底简单映射**：对明显的「端口/服务 → 工具」映射使用表驱动或规则引擎，不必每次都请求 LLM 决策，例如：
  - 80/443 端口开放 → 必选 Web 类扫描（二选一：nuclei/xray 等）；
  - 服务 banner 命中特定框架版本 → 直接挂载对应 PoC 模板。
- **LLM 专注复杂策略**：仅在需要综合多源信息（多端口/多资产/历史结果）进行权衡时调用 LLM Decision Engine，减少 token 消耗与延迟。

### 12.4 工具参数模板化

- 为常用 Skill 预先定义**参数模板**（Template），如：
  - `nmap_default_recon`：`-sV -sC -Pn` + 常用端口集；
  - `nuclei_web_high_risk`：高危 Web 模板集合；
  - `sqlmap_basic`：常规注入检测参数。
- LLM 决策时仅选择 **模板 ID + 少量覆盖字段**（如目标 URL / 端口），由 Skill Executor 在执行前将模板展开为完整命令：
  - 提升稳定性（避免 LLM 拼错命令）；
  - 减少 prompt 复杂度与决策空间。

### 12.5 缓存与结果复用

- **资产级缓存**：对可长期复用的结果（如 DNS 解析、子域名列表、少变动的端口信息）进行缓存，按目标或资产维度设置合理过期时间：
  - Orchestrator 在生成侦察任务前先查询缓存，命中时可跳过重复扫描或降低扫描强度。
- **任务级复用**：在同一任务内，对已完成的子任务结果（如特定 URL 的 nuclei 扫描）做去重与引用，而不是重新执行工具。

### 12.6 多工具交叉验证与重复控制

- **关键漏洞双工具验证**：
  - 对高危或影响较大的发现（如 RCE、未授权访问），策略上要求使用两种不同工具/方法进行验证，例如：
    - nuclei 模板 + xray；
    - PoC 脚本 + Metasploit 模块。
  - 仅当多工具结果一致时，将漏洞标记为「已确认」，否则为「待人工复核」。
- **执行次数与重复防控**：
  - Orchestrator 或管控层按 `(task_id, skill_id, target, params_hash)` 维护执行计数：
    - 对关键类 Skill 要求「至少两次」验证；
    - 设置最大执行上限（如同一组合最多 2～3 次），超过则不再下发相同调用。
  - 重复防控模块从 Trace 中计算「行动指纹/相似度」，在 LLM 输入中显式注入「重复警告」，降低无效重试概率。

### 12.7 结果解析与两层语义理解

- **规则优先的结构化解析**：
  - 对常见工具输出（Nmap、Nuclei、SQLMap、Metasploit 等）使用正则或脚本进行结构化抽取，生成 `parsed_artifacts`（端口/服务/CVE/风险等级等）。
- **LLM 辅助高层语义**：
  - 仅将结构化结果 + 关键片段摘要交给 LLM，用于：
    - 综合多个工具结果做风险评估；
    - 生成人类可读的描述与修复建议；
    - 进行阶段级或任务级总结。
  - 避免直接将大段原始输出整体喂给 LLM，降低 token 与误解风险。

### 12.8 优先级模型与攻击路径优化

- 在 Target Context 中为每个资产/端口/服务计算简单的**优先级分数**，参考：
  - 资产重要性（外网边界、关键业务、内网核心等）；
  - 服务敏感度（RDP/SSH/数据库/管理后台等）；
  - 已知漏洞数量与风险等级。
- Orchestrator 与 LLM 决策时显式使用该优先级：
  - 在同等资源与时间约束下，优先调度高优先级目标与路径；
  - 攻击路径构造时优先选择「高价值资产 → 已知弱点」的路径组合。

### 12.9 指标监控与策略迭代

- 基于 Trace 与报告输出，定期统计以下指标：
  - 单目标平均工具调用次数；
  - 工具失败/超时率；
  - 单目标平均耗时；
  - 漏洞发现率、误报率、复核通过率。
- 将上述指标与不同策略配置（如是否启用双工具验证、是否启用某些缓存或并行策略）进行对比，形成**可迭代的调优闭环**，为后续版本提供量化依据。

### 12.10 经验策略层与自适应优化

- **经验策略层（Policy Store）**：
  - 在 Orchestrator 之上增加轻量的经验策略层，维护针对不同场景的历史统计信息，例如：
    - 在何种目标/服务/阶段下，哪些 Skill/工具组合成功发现了真实漏洞；
    - 哪些组合长期产出较低或误报较多。
  - 可实现为独立的数据表或服务（Policy Store），按 `target_type / service / phase / skill_id` 等维度记录成功率、调用次数等统计字段。
- **经验驱动的优先级与模板选择**：
  - Orchestrator 在每次调用 LLM Decision Engine 前，从 Policy Store 读取当前场景下各 Skill 或模板组合的推荐权重：
    - 将这些权重作为额外字段传入 LLM 输入；
    - 或直接在本地对候选 Skill/模板按权重排序，过滤掉历史表现极差的选项。
  - 这样 LLM 决策不再是「从零开始」选择，而是带有「经验先验」，优先考虑历史上更有效的组合。
- **经验的产生与更新**：
  - 任务结束后，通过 Trace 与最终报告自动回收经验：
    - 对被标记为「已确认」的漏洞，反向更新对应 Skill/组合在该场景下的成功率；
    - 对多次执行但无有效发现的组合，降低其权重。
  - 更新策略可采用简单的滑动平均或加权计数，无需引入复杂的在线强化学习算法，即可实现「越用越聪明」的经验积累。
- **与 LLM 与规则的关系**：
  - 经验策略层不会替代现有规则与 LLM 决策，而是作为「第三信号源」：
    - 规则提供硬约束（如端口→必跑工具映射）；
    - LLM 提供复杂场景下的综合判断；
    - 经验策略提供「现实世界效果」的反馈与先验，三者结合形成更稳健的决策体系。
